## 2. Operating principle

Do not treat the stack as bookmarks.

Treat it as a business operating system.

Every repository should be assigned to one of four states:

1. `USE_NOW` — directly supports current work and should be activated immediately.
2. `STUDY_LATER` — valuable reference, but not urgent.
3. `ARCHIVE_REFERENCE` — keep for knowledge, but do not operationalize now.
4. `REVIEW_FIRST` — security, privacy, license, or maintenance risk must be checked before use.

## 5. Folder governance standard

Every active project repository should use this minimum structure where relevant:

```text
/docs
/data
/assets
/templates
/prompts
/workflows
/scripts
/reports
/exports
/README.md
/STATUS.md
/RISK_REGISTER.md
```

## 6. Required files for serious repos

Each active repo should eventually have:

- `README.md` — what the repo does
- `STATUS.md` — current operating status
- `ROADMAP.md` — next milestones
- `RISK_REGISTER.md` — key risks and mitigations
- `LICENSE_REVIEW.md` — license and commercial-use notes
- `.gitignore` — blocks local secrets and generated junk
- `.env.example` — safe example only
- `CHANGELOG.md` — what changed and when

## 9. Operating rule

A repo is not useful because it exists.

A repo is useful only when it is connected to an output, a workflow, a dashboard, a document, a product, a client, or a decision.

This stack must therefore be judged by execution, not collection size.
